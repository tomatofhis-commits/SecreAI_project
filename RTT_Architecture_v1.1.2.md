# RTT (Real-Time Translator) 統合アーキテクチャ詳細仕様 [v1.1.4 安定化・完全版]

## 1. 処理サイクルの全容 (The Loop)
RTTは **約6FPS (0.16s間隔)** の高速ループで動作し、1回のキャプチャ画像に対して「既存表示の維持」「座標の追従」「新規テキストの発見」を同時に行います。

### 1.1 監視ループと事前判定 (Pre-check) [v1.1.2 強化]
バックグラウンドでの重いOCR処理を起動するかどうかの「門番」です。
- **ピクセル差分判定 (L1-Norm)**: 64x36 縮小画像において、全画素の絶対値差分合計を計算。輝度フリッカーに強く、純粋な絵の変化を捉えます。
- **エッジ優先インテリジェント・フロー**: 単なる閾値判定ではなく、文字の形状（エッジ）を主役とした優先順位で判断します。
    1. **エッジ激変 (10%以上)**: 内容変化が確実なため、ピクセル変化に関わらず **即実行**。
    2. **エッジ安定 (2%未満)**: 背景が動いていても文字の形は不変とみなし、**即スキップ**。
    3. **ピクセル安定**: エッジ変化が中程度の場合のみ、ピクセル差分（標準2400）を確認し、静止していれば **スキップ**。
    4. **強制更新 (5秒)**: 静止画状態でも誤認識放置を防ぐため強制実行。
- **効果**: 3D背景の動きやスクロールを「背景の動き」として正しく無視し、文字の増減のみに反応できるようになりました。

## 2. ハイブリッドOCRパイプライン
高精度と低負荷を両立するため、二段階のエンジン連携を行っています。

### 2.1 索敵 (Scouting Phase)
- **WinRT OCR**: OS標準のエンジンを使い、画面全体のテキスト位置と内容を高速に取得。
- **PaddleOCR (rec=False)**: 文字認識を行わず、枠（矩形）の検出のみを高速に実行。
- **目的**: どこに文字があるかという「座標情報」の網羅。

### 2.2 統合 (Merge Phase)
- WinRTの結果とPaddle(Scouting)の結果を **IoU (Intersection over Union)** で突合。
- 重複する枠を整理し、読み取り対象となる `master_rects` を確定。
- **原則**: WinRT OCR からは **「矩形（枠）の情報」のみを抽出** し、読み取られたテキスト内容は一切使用せず、即座に破棄します。ノイズフィルタリングや翻訳の判定には、後の精読フェーズで得られる PaddleOCR の結果のみを使用します。

### 2.3 精読と細分化 (Refine & Subdivide Phase) [v1.1.2 最適化]
- 確定した各 `master_rect` に対して、**PaddleOCR によるフル認識 (rec=True)** を実行。
- **インテリジェント・パーティショニング**: 物理的な距離による機械的な細分化を抑制し、レイアウトの意味に基づいた分割を行います。
    - **PaddleOCR-First**: 基本的な「行」や「段落」のグループ化は PaddleOCR の内部解析能力に一任し、文章が不自然に細切れになるのを防止します。
    - **フォントサイズ判定 (主軸)**: 同一枠内に、文字の高さが **1.5倍** 以上異なる要素（例：巨大なタイトルと小さな説明文）が混在する場合のみ、別個のUIパーツとして分割します。
    - **巨大な空白の分離 (フォールバック)**: PaddleOCR が同一ブロックと誤認した場合の保険として、文字高の **4.0倍** 以上の極端な空白がある場合のみ、物理的に分割します。
- **目的**: 文章の文脈（コンテキスト）を維持しつつ、異なる役割を持つUI要素（ボタン、ステータス、チャット欄など）をそれぞれ最適なサイズの翻訳枠で囲うこと。

## 3. 多段ガードレール (OCR Guardrails)
ハルシネーション（誤読）やゴミ文字を排除するための物理フィルタです。

### 3.1 記号・部首フィルタ
- **記号率**: 12文字以下の短文で25%以上、それ以外で35%以上の記号を含む場合に破棄。
- **部首判定**: PaddleOCR特有の「意味のない部首の連続」を検知し破棄。

### 3.2 言語横断 Unicode ブロック判定 [v1.1.2 修正]
- **挙動**: 設定画面で有効にされている **すべての言語** の文字種を許可リストとして保持。
- **判定**: 許可リストに含まれない文字（ゴミ文字）の割合が **40%** を超え、かつ確信度が **0.85** 未満の場合に「不適合」としてスキップ。

## 4. 堅牢性と安全性の強化 [v1.1.2 導入]
アプリケーションの致命的なクラッシュやフリーズを根絶するためのスレッド安全設計です。

### 4.1 スレッドセーフな初期化 (Worker Thread Initialization)
- **課題**: PaddleOCR の初期化（CUDA/ONNXコンテキストの構築）を一時的なスレッドや別スレッドで行うと、コンテキストの破棄やスレッド跨ぎのアクセスにより `Unhandled Python exception` でアプリが強制終了していました。
- **実装**: PaddleOCR の初期化を、起動時から常に存在し続ける **専用のワーカースレッド (`_ocr_worker`)** 内で `preload()` 実行するように設計変更。
- **効果**: CUDAコンテキストが生成されたスレッド内に固定されるため、グラフィックドライバや実行ランタイムとの競合が解消され、安定性が劇的に向上しました。

### 4.2 Flask API と UIスレッドの分離 (Signal-based GUI Safety)
- **課題**: 通信用の Flask スレッドから直接 GUI 要素（翻訳枠の表示・非表示など）を操作すると、PyQt のアクセス違反により即座にクラッシュしていました。
- **実装**: Flask スレッドからは **PyQt Signals (合図)** を送るのみに留め、実際の GUI 操作（翻訳の開始・停止・非表示など）は必ずメインスレッドのイベントループ内で実行されるように通信経路を再構築しました。

### 4.3 詳細なクラッシュ・トラッキング (Global Exception Hook)
- **実装**: `sys.excepthook` をオーバーライドし、PyQt のスロット内等で発生した「中身の見えない例外（Unhandled Exception）」をキャッチして、詳細なトレースバックをコンソールに出力する仕組みを導入。開発時およびユーザー環境での原因特定を迅速化しました。

## 5. 二層式トラッキング・アーキテクチャ [v1.1.2 刷新]
「枠の乖離」による不安定さを解消し、低負荷で高精度な追従を実現する新方式です。

### 5.1 Step 3（粗枠）と Step 4（詳細枠）の分離保存
- **概要**: 検出用の「親枠（粗枠）」と、精読後の「表示用詳細枠」を分けてステータスに保持します。
- **効果**: 監視ループにおいて「前回見つけた粗枠」と「今回の粗枠」を比較（リンゴとリンゴの比較）できるようになり、文字読み取りによる枠サイズの微変に伴う誤判定（ゴースト化・消去）を根絶しました。

### 5.2 OpenCV テンプレートマッチング追従 [v1.1.4 強化]
- **概要**: 6FPSの監視ループで、前回取得した「粗枠内のピクセル画像」を用いて、現在の画面内を OpenCV の `cv2.matchTemplate` で高速探索します。
- **テンプレートの永続保持化 [v1.1.4 修正]**: 翻訳結果の反映処理（`_on_single_translation_done`）において、それまで保持していた追従用画像データ `step3_crop` と `parent_rect` が上書きにより消失するバグを修正。翻訳完了後もシームレスに位置追従が継続するように改善しました。
- **探索範囲（マージン）の拡張 [v1.1.4 強化]**: スクロールや高速なオブジェクト移動に対応するため、テンプレートの探索マージンを `±20px` から **`±50px`** へ大幅に拡張。急激なフレーム内移動でも見失わずに追従が可能になりました。
- **dx/dy 移動量計算**: 探索により見つかった位置から移動量 `(dx, dy)` を算出し、粗枠と詳細枠の両方に加算します。
- **メリット**:
    1. **負荷ゼロ**: 1回あたり 1ms 未満で完了するため、CPU負荷を増やさずに 6FPS の滑らかな追従が可能です。
    2. **精度**: 1ピクセル単位の移動も正確に捉え、ゲームや動画内の字幕移動にピタリと吸い付きます。
    3. **安全性**: UIスレッドで重いOCRを回す必要がなくなるため、フリーズやクラッシュの危険がありません。

## 4. 座標トラッキングと表示維持 [v1.1.2 大幅刷新]

### 4.1 IDトラッキングと原文の尊重
- **ID継承判定**: 新旧の矩形が IoU > 0.1 かつ **テキスト類似度が 0.8 以上** の場合のみ同一IDとして継承。
    - **[旧仕様(修正済み)]**: 類似度 0.4 という極めて緩い条件でIDを継承させていた。
- **原文ハイジャックの廃止**: IDが継承された場合でも、**新しくOCRしたテキストを優先** します。
    - **[旧仕様(修正済み)]**: IDが継承されると、新しく読み取った文字を捨てて「古い方の文字」に強制的に書き換えていた。これが「文字が変わっても翻訳が変わらない」最大の原因であったため、完全に廃止。
- **座標重複の即時排除 (IoA/IoU判定)**: 新しいIDのチャンクが既存の表示と重なった場合、以下の基準で古い方を「残骸」とみなし即座にパージ（削除）します。
    - **IoA (Intersection over Area) > 0.5**: どちらか一方がもう一方に50%以上覆われている場合、サイズに関わらず比較対象とします。
    - **細分化対応**: 大きな古い枠の中に小さな新しい枠が現れた場合、新しい方が「高品質」または「内容変化」と判定されれば、古い巨大な枠を優先的にパージします。
    - **[旧仕様(修正済み)]**: `IoA > 0.4` かつ `area1 < area2 * 0.5` の場合、新しい小さな枠を「ノイズ」として捨てていた。これが細分化時に古い表示が張り付く原因であったため廃止し、新しい情報による「追い出し」を優先するロジックに変更。

### 4.2 猶予期間と消去ロジック
- **猶予期間 (Grace Period)**: 
    - **現行**: **1.0秒**。レスポンス向上のため短縮。
    - **[旧仕様(修正済み)]**: 3.0秒。表示が消えるのが遅すぎ、古い翻訳が残り続ける原因となっていた。
- **自動クリア**: 暗転やシーンチェンジ（エッジ密度不足）を検知した場合、即座に全表示と履歴をクリア。

## 5. 表示安定化と更新判定 (Stability Control) [v1.1.2 新規]

### 5.1 コンテンツ変化の2ストライク制
同一箇所のテキストが変化した際、チラつきを抑えつつ高速に更新するための判定です。
- **判定**: 類似度が **0.6未満** の変化を検知した場合、内部的にストライクをカウント。
- **確定**: **2回連続** で変化を検知した瞬間に、古い翻訳表示を抹消し、新しいテキストでの再翻訳を開始。
- **キャッシュ・ファストパス (即時パージ) [v1.1.2 強化]**:
    - **例外**: 以下の場合はストライク（2回待ち）を無視し、即座に古い表示を削除して新しい内容を反映させます。
        - 読み取った新しいテキストが既に **キャッシュに存在する場合**（正しい翻訳を即座に出せるため）。
        - 類似度が **0.25未満**（明らかに全く別の文章に切り替わった場合）。
- **高品質優先 (High-Score Override)**: 新しいOCRスコアが既存より **0.5以上** 高い場合も即座に更新。

### 5.2 ストライク・リセットの厳格化
- **条件**: 類似度が **0.95以上**（ほぼ完全一致）であるフレームを検知した場合のみ、蓄積された変化ストライクを 0 にリセットします。
- **目的**: OCRのノイズによって「内容が変わった」という判定が途中で取り消されるのを防ぎ、確実に更新を完遂させること。

## 6. キャッシュ検索と曖昧一致 (Cache Strategy) [v1.1.2 強化]

### 6.1 全長における正規化一致の解禁 [v1.1.2 強化]
- **対象**: **すべてのテキスト**（短文・長文を問わず）。
- **挙動**: `normalize_text` を通した後の正規化一致（Stage 2）を全テキストで許可。
- **目的**: OCRの微細な揺れ（末尾のドット `.` やカンマ `,` の誤認）が発生しても、キャッシュにヒットさせて再翻訳（と表示のチラつき）を阻止。

### 6.2 強力な記号ノイズ除去
- **正規化の強化**: `normalize_text` において、OCRノイズになりやすい記号（`. , : ; ! ? ' " ` ^ ~` など）を積極的に除去対象に追加。
- **曖昧一致 (長文のみ)**: 20文字以上の長文に対しては、引き続き閾値 **0.99** での曖昧検索を併用。


## 7. 高解像度キャプチャ・エンジン (High-DPI Capture) [v1.1.2 強化]
高解像度ディスプレイ環境において、OCR精度を `mss` 同等以上に引き上げつつ、オーバーレイの除外を両立させるための仕組みです。

### 7.1 DPI-Aware 物理ピクセルキャプチャ
- **概要**: Windowsの拡大設定（125%、150%等）を自動検知し、論理座標ではなく「物理ピクセル」単位でキャプチャを行います。
- **実装**: `GetDpiForWindow` を使用してウィンドウごとのDPI倍率を取得。
- **BitBlt (高画質モード)**: デスクトップDCから物理座標で直接切り抜くことで、ボヤけのない鮮明な画像を取得。
- **PrintWindow (高画質モード)**: メモリ上のビットマップを物理サイズで確保し、OSに実寸大での描画を要求。
- **ジッター防止 (Rounding)**: 座標計算時に `round()` による丸め処理を徹底し、1ピクセル未満の浮動小数点誤差による「画面の震え（ジッター）」と、それに伴う不要なOCRトリガーを根絶。

### 7.2 三段階のキャプチャ・フォールバック [v1.1.4 最適化]
- **DXCAM (WGC) の標準化 [v1.1.4 導入]**: CPU負荷低減と高画質キャプチャの両立のため、**高速キャプチャ (WGC / DXCAM)** をデフォルトのキャプチャエンジンとして固定・安定化させました。
- **1. BitBlt (レイヤード除外)**: レガシー環境用の代替最優先。
- **2. PrintWindow (レイヤード除外)**: BitBltで真っ黒になる窓への代替。
- **3. 互換モード (mss)**: 上記で「対象ウィンドウが見つかりません」となる場合の最終手段。
    - **仕様**: `mss` は画面ピクセルを直接撮るため、UI側の「ステルスモード（キャプチャ除外設定）」を併用してクリーンな画像を得ます。

## 8. 強力な重複排除とノイズフィルタ [v1.1.2 導入]
OCRがゴミを拾い続けたり、古い翻訳がループして表示される現象を根絶するための防衛線です。

### 8.1 四段階のテキスト・ブロック
1. **バッチ内重複排除**: 同一フレーム内で見つかった同じテキストは1つに統合。
2. **翻訳中 (Pending) ブロック**: 現在LLMが翻訳中のテキストと同じものは、追加でキューに入れない。
3. **短寿命履歴ブロック (TTL: 2s)**: 過去2秒間に処理したテキストと同じものは、OCRで見つかっても無視。
4. **ゴミ判定**: 記号のみの1文字、または 8x8 ピクセル未満の極小チャンクを物理的に除去。

### 8.2 チャンク数の動的制限
- **概要**: 1フレームあたりの最大読取・翻訳数を **30個** に制限。
- **優先順位**: 30個を超えた場合、「文字数が多いもの（長文）」「面積が大きいもの（メインテキスト）」を優先してキューに送り、背景の細かなノイズを自動的に切り捨てます。
